# FiTuna 사용 시나리오

실측 수치가 붙은 시나리오는 실제 측정값(`docs/RESULTS.md`)이고, 그 외는
동작 방식 설명이다. 수치를 지어내지 않는다 — 그게 이 도구의 존재 이유다.

---

## 시나리오 1 — 노트북에서 로컬 코딩 어시스턴트 돌리기 (실측 기반)

**상황.** 인공지능학과 학생. M3 Pro 맥북(통합메모리 18GB)에서 API 비용 없이
로컬 LLM을 코딩 보조로 쓰고 싶다. 체감상 답변이 술술 읽히려면 30 tok/s는
나와야 하고, 코드 품질이 눈에 띄게 나빠지는 건 싫다.

**기존 방식.** HuggingFace에서 Q4? Q5? Q8? 커뮤니티 글마다 말이 다르다.
하나 받아서 돌려보고, 느리면 다시 받고, `-ngl` 값을 바꿔가며 수동 벤치마크.
반나절 소모.

**FiTuna.**

```bash
fituna run --model Qwen3-4B-Instruct-2507-F16.gguf \
  --target-tps 30 --max-quality-loss 5 --ctx 4096 \
  --quant Q8_0,Q6_K,Q5_K_M,Q4_K_M --ppl-chunks 32 \
  --wikitext wikitext-2-raw-test.txt --out ./out --resume
```

**실측 결과 (M3 Pro).** "당연히 최고"라던 Q8_0은 24.22 tok/s로 탈락,
Q6_K 28.48 탈락, Q5_K_M은 29.59로 **0.41 차이** 탈락 — 사람이 감으로
골랐으면 여기서 멈추고 목표 미달인 채 썼을 것이다. 최종 답은
**Q4_K_M을 GPU 레이어 33개만 올린 구성, 30.81 tok/s, 품질손실 1.73%**.
출력된 `llama-cli` 커맨드를 그대로 복사해 바로 사용.

## 시나리오 2 — "이 컴퓨터로 되긴 하나?" 현실 확인 (exit code 3)

**상황.** 구형 데스크톱이나 미니PC 사용자. 로컬 LLM을 돌리고 싶은데 이
하드웨어로 목표 속도가 나오는지조차 모른다.

**FiTuna.** 목표를 넣고 돌리면, 모든 후보가 미달일 때 조용히 실패하지 않고
**종료 코드 3 + 가장 근접한 best-effort 구성**을 알려준다.

**실측 예 (SmolLM2-135M, 목표 300 tok/s).** 전 후보 미달 → "최선은 Q6_K
249.50 tok/s"를 33.6초 만에 보고. 사용자는 세 가지 중 선택하면 된다:
목표를 낮추거나, 더 작은 모델을 쓰거나, 하드웨어를 바꾸거나 — **찍어보기
전에 답을 안다.**

## 시나리오 3 — 연구실/소규모 팀의 온프레미스 서빙 구성

**상황.** 연구실 공용 워크스테이션에 사내용 LLM을 띄운다. 요구사항은
"동시 사용자 응답이 견딜 만한 속도(예: 20 tok/s)" + "품질 저하 3% 이내".
장비는 한 대뿐이라 벤치마크로 장시간 점유하기 부담스럽다.

**FiTuna.**
- 탐색 비용은 한 번만: 결과가 `.fituna_cache.sqlite3`에 저장되고, 같은
  모델·하드웨어·llama.cpp 빌드 조합이면 `--resume` 재실행이 **1초 미만**
  (실측 0.88초)에 같은 답을 재현한다. 캐시 파일을 팀에 공유하면 동일 장비
  구성원은 탐색 자체를 건너뛴다.
- 캐시 키에 llama.cpp 빌드 버전이 포함되므로, 백엔드를 업그레이드하면
  낡은 측정값이 몰래 재사용되는 일이 없다 — 자동으로 재측정된다.
- 요구사항이 바뀌면(20 → 25 tok/s) 이미 측정된 후보는 재활용되고 부족한
  측정만 추가로 돈다.

## 시나리오 4 — 야간 배치로 "내 장비 프로파일" 만들기

**상황.** 낮에는 장비를 써야 해서 벤치마크를 돌릴 수 없다.

**FiTuna.** 퇴근 전에 후보를 넓게 걸어 실행해두면(`--quant`에 6개,
`--ctx 8192,4096`), 다음날 캐시에 모델×양자화×ngl×ctx 실측 표가 쌓여
있다. 이후에는 어떤 목표치를 넣어도 캐시에서 초 단위로 답이 나온다 —
목표 40으로 물었다가 30으로 다시 물어도 재벤치마크 없음(실측: 목표 변경
재실행 수 초 내). 도중에 전원이 나가도 `--resume`이 끝난 지점부터
이어간다(양자화 파일 생성은 원자적 쓰기라 깨진 파일이 남지 않는다).

## 시나리오 5 — AI 코딩 에이전트의 실측 백엔드 (로드맵: MCP)

**상황.** 사용자가 AI 에이전트에게 "내 컴퓨터에서 돌릴 로컬 모델 추천해줘"
라고 묻는다. 지금의 챗봇은 하드웨어 스펙 텍스트를 보고 **추측**으로
답한다 — 그리고 시나리오 1이 보여주듯 그 추측은 0.41 tok/s 차이로도
틀린다.

**FiTuna(MCP 확장 후).** 에이전트가 FiTuna MCP 서버를 호출하면 추측 대신
그 자리에서 실측된 구성을 돌려받는다. 에이전트의 답변이 "아마 Q5면 될
거예요"에서 "방금 측정했습니다: Q4_K_M ngl=33, 30.81 tok/s"로 바뀐다.

## 공통 전제와 한계 (정직 고지)

- FiTuna는 추론·양자화 연산을 직접 하지 않는다. llama.cpp 바이너리가
  필요하다 (`brew install llama.cpp` 또는 소스 빌드).
- 품질 지표는 wikitext-2 perplexity 단일 축 — 범용 프록시일 뿐, 한국어·
  코딩 등 도메인별 품질을 보장하지 않는다.
- 탐색은 후보 양자화 파일을 전부 생성한다(4B 모델 4개 후보 기준 디스크
  약 12GB). 파일은 재실행 시 재사용된다.
- 벤치 수치는 서멀 상태에 민감하다. 목표와 수 tok/s 이내로 아슬아슬한
  판정은 장비가 평상시 상태일 때 재실행 권장 (`docs/RESULTS.md`의 변동성
  실측 참고).
